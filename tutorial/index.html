Reactive chat application with Play, Akka Distributed Data and Websockets

The purpose of this activator template is to illustrate how a horizontally scalable, reactive architecture with websockets can be implemented with use of WebSockets and Akka with Akka Distributed Data.

Challenge

WebSockets are a protocol which enables bi-directional communication between user's browser and the server. With use of websockets you can send messages to the server and receive them independently from HTTP request/response cycle. Websockets are a powerful protocol and are reshaping the web by enabling truly responsive and real-time user experience, eliminating the need for extra browser plugins (eg. Applet, Flash) or workarounds (remember Comet?). Common use cases of such communication mechanism include:

  • Chats, instant messangers
  • Bidding platforms
  • Multi-player games
  • Various forms of on-line data broadcasting
While the answer is quite straight forward how to implement async communications with HTML5 on the browser side, you have many options on the backend. This tutorial decribes one good option how such scalable backend can be implemented. By going through this tutorial you will learn how to:
  • Create asynchronous message channels in your web pages with use of websockets.
  • Tie websocket connections to a reactive, scalable backend.
  • Implement the backend in a distributed, message driven manner where messages are shared between all cluster nodes.
Read on.

Solution

You might wonder about best patterns and blend of technologies to implement such communication. This application template aims to be a blueprint of such implementation based on the core elements of the Typesafe Reactive Platform:
  • Play Framework
  • Akka
  • Scala

Also an interesting challenge emerges when you decide to scale out your backend by adding server instances and require to maintain message distribution between these nodes ensuring their delivery to all clients, regardless of their connection to one particular cluster node. It turns out that such scenario can be handled by Akka Distributed Data by enabling the distribution of messages among server instances within the cluster.

Let's build an example application that demonstrates such use case. We chose to exhibit this architecture by implementing a chat web app scenario.

Frontend

The web part of the application includes two web pages. The first one is home/index page. After opening a home/index page you will see a simple form to define your nickname. To go further to chat page, you should fill out the form and submit it. To go back to home page, you should press the `leave chat` link to exit the chat page. On the chat page users communicate by the following scenario: users create topics, subscribe to them and, if they happen to be subscribed to the same topic, they can chat. All chat messages are transferred through an active web socket.

The whole client side chat logic is implemented within chat.coffee file, to keep our code compact. Moreover, we use sbt-coffeescript plugin (take a look at : plugins.sbt) to compile CoffeeScript sources to JavaScript. Examine the chat.coffe file to learn how:

  • WebSocket object is instantiated
  • messages are marshaled and unmarshaled with use of JSON object
  • messages are sent upon form submit
  • messages are received from the server by onmessage callback of the WebSocket object

Backend

You would rather run cluster nodes not only in separate JVMs but also on different physical machines in production. This can be excessive for testing and experimenting. To simplify cluster setup, we will simulate it by starting three actor systems on different ports. The configuration of actor systems is located in application.conf file.

Nevertheless, you can adjust the configuration if your preference is to run them on different machines. This can be done by redefinition of the akka.cluster.seed-nodes section in the aforementioned application.conf coniguration file.

To demonstrate the communication between nodes we will start three Play app instances, as reflected in the application.conf file. Assuming that this file remains unchanged, we will start the first Play application node on port 9001 of your box. Port 2551 will be used for remote communication. The second one will be started on port 9002 with port 2552 for remoting, and the last node on port 9003 and port 2553 respectively. To achieve this aim you should run the following commands when using a Linux/Unix-like system:

  • ./activator "run -Dhttp.port=9001 -Dakka.remote.netty.tcp.port=2551"
  • ./activator "run -Dhttp.port=9002 -Dakka.remote.netty.tcp.port=2552"
  • ./activator "run -Dhttp.port=9003 -Dakka.remote.netty.tcp.port=2553"

Also some sort of load balancer in front of the cluster (eg. HAProxy) is required to distribute the traffic between nodes in a real world scenario.
Load balancer configuration is beyond the scope of this tutorial, but let us just note that care should be taken to configure the load balancer properly, to maintain active websocket connections in a multinode setup. More info on HAProxy configuration for websockets can be found here.

The mechanics

Please note that the /socket context in routes file points to controllers.Chat.socket controller method. Take a look how connections are handled with use of Play's WebSocket handler.
Each user has their own websocket open which is serviced by UserSocket.scala. The UserSocket receives play.api.libs.json.JSValue messages from the user's browser, parses it and eventually publishes a ChatMessage(topic: String, user: String, text: String) (or TopicNameMessage(topicName: String) - indicating that a new topic was created). Upon it's creation, a UserSocket registers to the akka.cluster.ddata.Replicator by sending a Subscribe message with GSetKey, initialized with the name "topics".

Distributed Data Types

In this tutorial we use two of the data types that Akka Distributed Data provides us with:

  • GSet. The "G" means "grow-only", we can't delete from it. We use GSets to store the list of topics and to store messages for each topic.
  • LWWRegister - stores one value. LWW means "last write wins". We use it to store the last message in each topic.
A bunch of distributed sets would be enough to build a chat, so why do we use LWWRegisters? The reason is that it simplifies the logic of the callback to the "new message" event. When GSet gets updated, it sends the whole set to subscribers, but we need only the new message, that's why we use LWWRegister.

Why not just use LWWRegisters? We need GSets to make new users able to retrieve the list of topics and the list of messages for the selected topic; to put it another way, we need GSets to store the history.

Let's consider the following scenario:

  1. First User checks in, creates two topics, writes a message in each of them and waits for the answer in the second topic.
  2. Second User arrives, goes to the second topic and chats with First User.
  3. Third User creates a topic, goes to the second topic, reads the conversation and writes "I'm glad I didn't miss anything".
Here is how distributed objects represent this process:

As we can see, the topics list is represented by a GSet, and each topic is represented by a GSet paired with an LWWRegister.

To publish the message we perform two operations:

  • Add a message to the topic's GSet.
  • Store the message in the topic's LWWRegister.
The corresponding code is a bit verbose, so let's first have a look at the simplified version of it: 
            replicator ! Update("the key to find the gset of the topic", ...) {
              // how to update the gset
            }
            replicator ! Update("the key to find topic's lwwregister", ...) {
              // how to update the register
            }

The code sends two Update messages to the replicator. When constructing the Update message, we initialize it with the key that the replicator will use to find the object, and with a function that does the updating.

Here is what the actual code looks like: 

            replicator ! Update(GSetKey[ChatMessage](topic), GSet.empty[ChatMessage], WriteLocal) {
              _ + chatMessage
            }
            replicator ! Update(LWWRegisterKey[ChatMessage](topic + "-lwwreg"), LWWRegister[ChatMessage](null), WriteLocal) {
              reg => reg.withValue(chatMessage)
            }

The DistributedData and the replicator Actor is used in the code to provide abstraction for message distribution between cluster nodes. It is a nice implementation of publish-subscribe communication style with distributed objects which uses actors Akka Routing under the hood to distribute messages in the cluster. The use of this feature is very simplistic since message passing is done same way as to local Actors - it is location transparent. One just has to subscribe to events to receive distributed messages. This way, with use of DistributedData there is no concern, or little about clustered environment upon message passing from programmer's point of view. In our template if you inspect the UserSocket Actor you will find that it subscribes to replicator : 

            val replicator = DistributedData(context.system).replicator
            ...
            replicator ! Subscribe(topicsKey, self)
When one of users creates a topic, his UserSocket will update a GSet that stores the topics. Since all users are subscribed to this event, the system will update this GSet for all users, and a new topic name will be shown in each user's browser page along with the "subscribe" button. When a user clicks "subscribe", his UserSocket will send a Subscribe message to replicator (it first sends the Get message to get the history from GSet and then, when history arrives, it subscribes to LWWRegister). After that a user will communicate with others through the mediator by sending Update that update topic's GSet and LWWRegister with ChatMessage messages. The received ChatMessage is serialized and sent to out.

For detailed information on this mechanism please check documentation on Akka web page.

A scenario of message distribution between nodes.

Let's play a scenario with six users - named A-F. The user A is connected to host on port 9001. Users B, C and D are connected to host on port 9002. The rest of them are connected to host on port 9003, as shown on the diagram below.

As decribed above the Akka Cluster feature is leveraged to implement the delivery of messages in a dispersed multi-node clustered environment. The steps below demonstrate how a chat message which is sent by user A from 9001 instance is broadcasted to all participants.

  1. user A defines and sends a message, the corresponding UserSocket receives it
  2. A's UserSocket parses the message and sends to node 9001's Replicator
  3. Replicator sends the message to all UserSocket cluster nodes (9001, 9002, 9003)
  4. eventually, each UserSocket sends a message to the browser where it is served to the client.

Testing

You can also take a moment to examine the approach to testing, actors in particular. ChatMessageSpec test verifies serialization to JSON of ChatMessages. UserSocketSpec tests actor behaviour. The tests are based on Specs2 and Akka akka-testkit facilities including the TestProbe mocking class. The tell method is used to pass messages with second parameter to simulate the sending actor. The expextMsg method is used to assert message flow within the actor system in accordance with specification.

Summary

In this application template we have demonstrated how a message driven web application using websockets can be implemented with Akka and scaled out thanks to Akka Cluster at it's core. The application is a good starting point if you want to build a scalable and resilient cluster of websocket services. By adding cluster management facilities, whether simple load balancing (eg. with use HA Proxy) or a complete (micro)service manager solution like ConductR or Kubernetes if you like Docker, you will get a complete, managed, scalable and reactive cluster of websocket services communicating with the frontend.
